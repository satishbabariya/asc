# RFC-0003 Completion — Design Spec

**Date:** 2026-04-11
**Goal:** Close 2 remaining feasible gaps in RFC-0003 to bring coverage from ~88% to ~95%.

---

## Gap 1: Drop Destructor Dispatch

**Problem:** `own.drop` in OwnershipLowering calls `free()` directly without invoking the user's `Drop::drop()` method first.

**Fix:** In `OwnershipLowering.cpp`, when lowering `own.drop`:
1. Check if the dropped value's type has an associated drop function (mangled as `__drop_TypeName`)
2. If found, emit `func.call @__drop_TypeName(ptr)` before `free(ptr)`
3. If not found, just call `free()` as before

The drop function must be generated by HIRBuilder when processing `impl Drop for Type { function drop(...) { ... } }`. HIRBuilder should emit the drop method as a regular `func.func @__drop_TypeName` with the body from the impl block.

**Files:** `lib/CodeGen/OwnershipLowering.cpp`, `lib/HIR/HIRBuilder.cpp`

## Gap 2: Closure Captures for task.spawn

**Problem:** `task.spawn(worker)` passes `NULL` as the `void *arg` to pthread_create. The worker function has no access to captured variables.

**Fix:** In HIRBuilder's `task_spawn` handler:
1. If the closure argument is a `ClosureExpr` (not a bare function name), collect the captured variables
2. Allocate a closure struct on the heap: `{ capture1, capture2, ... }`
3. Store captured values into the struct
4. Pass the struct pointer as `void *arg` to pthread_create
5. In the wrapper function, load captures from the struct before calling the closure body
6. Free the closure struct after the wrapper returns

For the simpler case of bare function names (`task.spawn(worker)` where `worker` takes no captures), the current NULL-arg behavior is correct.

**Files:** `lib/HIR/HIRBuilder.cpp`

## Success Criteria

1. `impl Drop for T { function drop(...) { ... } }` — drop method called before free
2. `task.spawn(() => { use_captured_var(); })` — captured variables accessible in spawned task
3. All 173 existing tests still pass
4. New e2e tests for both features

## Out of Scope

- Constant folding (ConstantExpr) — arith.constant at HIR level is sufficient
- Overload resolution ranking — basic function lookup works
- Wasm-native atomics — C runtime with GCC builtins is functionally correct
